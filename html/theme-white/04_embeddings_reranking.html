Recap: Where We Are

The Bag-of-Words Legacy enables web-scale retrieval but lacks semantic depth.
  • Efficiency: Inverted indexes enable sub-second search.
  • Scoring: BM25 provides a robust lexical baseline.
  • Expansion: RM3 bridges some lexical gaps via feedback.

The Persistent Problem

Methods rely on exact term matches:
  • No semantics: car problems $\neq$ automobile issues.
  • Polysemy: bank (financial) = bank (river).

The Representation Gap

Lexical matching requires exact term overlap to estimate relevance.
  • Sparse: Each dimension corresponds to a vocabulary word.
  • Dense: Dimensions represent latent semantic features.
  • Goal: Learn embeddings where closeness $\approx$ same meaning.

The Representation Gap (Figure)

The Distributional Hypothesis

Core Principle (Firth, 1957)

``You shall know a word by the company it keeps.''
Example: What is a ``tezguino''?
  • ``A bottle of tezguino is on the table.''
  • ``Everybody enjoys tezguino.''
  • ``We brew tezguino from corn.''
Inference: It's an alcoholic drink. Context reveals meaning!

Word2Vec: Words as Vectors

The Skip-gram model learns word meaning from local context windows.
  • Hypothesis: Words in similar contexts have similar meanings.
  • Task: Predict surrounding context tokens from a target word.
  • Objective: Weights in the hidden layer become the embeddings.

Word2Vec: Words as Vectors (Figure)

Word2Vec: Semantic Properties

Semantic Properties

$\vec{\text{king}} - \vec{\text{man}} + \vec{\text{woman}} \approx \vec{\text{queen}}$
Why this matters for IR:
  • Vectors capture relational structure.
  • Semantic similarity $\approx$ vector proximity (cosine similarity).

Word2Vec: Relevance to IR

Vector proximity enables matching beyond exact keywords.
  • High Sim: $\cos(\vec{\text{car}}, \vec{\text{automobile}}) \approx 0.7$.
  • Low Sim: $\cos(\vec{\text{car}}, \vec{\text{banana}}) \approx 0.1$.
  • Recall: Enables finding relevant docs with zero term overlap.
The Naive Approach: $\vec{d} = \frac{1}{|d|}\sum_{w \in d} \vec{w}$
  • Mechanism: ``the cat sat on the mat'' $\to \text{Avg}(\vec{\text{the}}, \vec{\text{cat}}, \ldots)$
  • Pros: Simple, fast, significantly better than BoW for vocabulary mismatch.

Query Processing with Word Embeddings

How can we build an IR system using word embeddings?
  • Approach: Represent both the query $q$ and the document $d$ as vectors in the same $d$-dimensional space.
  • Mechanism: Use a pre-trained embedding table to look up vectors for each word.
Usage in IR:
  • Represent: $\vec{q}$ and $\vec{d}$ are derived from word vectors (e.g., via averaging).
  • Rank: Documents are ranked by vector similarity: $\text{score}(q, d) = \cos(\vec{q}, \vec{d})$.

So what is the problem?

The Static Vector Bottleneck

Each word has exactly one vector regardless of its context.
  • bank (river) = bank (financial).

Critical Failures

  • Loses word order (``dog bites man'' = ``man bites dog'').
  • Polysemy unresolved.

Static vs. Contextualized Embeddings

The 2018 evolution moved IR from fixed word vectors to dynamic, context-aware representations.
  • Static: Word2Vec/GloVe assignment is fixed (ignores neighbors).
  • Contextualized: BERT embeddings adapt to local word usage.
  • Polysemy: Distinguishes between ``bank'' (finance) and ``bank'' (river).

Why Transformers?

Attention mechanisms enable models to process entire sequences in parallel with global context. They came after Recurrent Neural Networks (RNNs) and LSTMs that solved the sequence problem but were limited
  • Parallelism: Processes all tokens simultaneously (unlike sequential RNNs).
  • Recency: Overcomes "forgetting" in long documents.
  • Scale: Efficiently trained on massive web-scale corpora.

The SOTA Shift

Transformers provide the foundation for every modern high-accuracy retrieval and re-ranking system today.

Transformer Architecture

Self-Attention: The Core Idea

Token representations are updated by "attending" to the most relevant neighbors.

Self-Attention: The Core Idea (Figure)

Self-Attention: The Mechanism

Interaction is computed via dot-products in Query, Key, and Value spaces.
  • Query ($\vec{q}$): ``What am I looking for in other tokens?''
  • Key ($\vec{k}$): ``What information do I offer to others?''
  • Value ($\vec{v}$): The actual information content to be aggregated.

The Attention Formula

\[ {Attention}(Q, K, V) = {softmax}({QK^T}{{d_k}}) V \]

Breaking Down the Formula

\[ {Attention} = {softmax}({QK^T}{{d_k}}) V \]
  • Dot Product ($QK^T$): Measures raw similarity between tokens.
  • Scaling ($\sqrt{d_k}$): Stabilizes gradients for deep training.
  • Processing: Softmax converts scores to weighting factors (sum to 1).

Worked Example: Information Flow

Sentence: ``retrieval models'' ($d_k = 2$)
  1. Similarity: $\vec{q}_{\text{retr}} \cdot \vec{k}_{\text{retr}} = 0.9$ (Self) $\vec{q}_{\text{retr}} \cdot \vec{k}_{\text{mod}} = 0.6$ (Neighbor)
  2. Weights: $\text{softmax}([0.9, 0.6]) \approx [\,\mathhl{0.57},\; \mathhl{0.43}\,]$
  3. Output: $\vec{h}_{\text{retr}} = 0.57 \vec{v}_{\text{retr}} + 0.43 \vec{v}_{\text{mod}}$

Interpretation

The representation of ``retrieval'' is now updated with 43% of the information from ``models''. Context is integrated!

Multi-Head Attention

Parallel attention heads learn different linguistic and semantic relationships.
  • Heads: BERT uses 8 heads per layer in parallel.
  • Variety: Heads learn distinct features (Syntax, Entities, etc.).
  • Robustness: Aggregating heads provides a richer representation.

Feature Extraction

Calculates 8 different embeddings per token, then concatenates them.

Positional Encoding

Since attention is permutation-invariant, we must inject word order manually.
  • Problem: Without positions, dog bites man = man bites dog.
  • Solution: Add a unique position vector $\vec{p}_i$ to each token $\vec{e}_i$.
  • Result: Model learns both identity and relative location.

Combined Input Representation

\[ {x}_i = {e}_{{word}_i} + {p}_i \]

The Transformer Encoder Block

Each encoder block combines parallel attention with per-token feed-forward processing.
  • Attention: Token interaction via Q, K, V.
  • Residuals: Prevents gradient signal decay.
  • Norm: Stabilizes flow through deep layers.

The Transformer Encoder Block (Figure)

What is BERT?

BERT (Bidirectional Encoder Representations from Transformers) is an encoder-only model.
  • Encoder vs.\ Generative:
  • Generative (e.g., GPT): Causal; predicts the next token.
  • Encoder (BERT): Bidirectional; sees the entire sequence simultaneously.
• Context Aware: Embeddings are computed based on all surrounding tokens. • Instance: BERT is a specific implementation of a Transformer Encoder. Encoder Representation

What is BERT? (Figure)

BERT Architecture: Core Components

The BERT architecture consists of three fundamental layers of processing.
  1. Embeddings Layer:
    • BPE / WordPiece: Splits tokens into sub-words (e.g., embedding $\to$ embed, ##ding).
    • Positional Encoding: Injects sequence order info.
  2. Multi-Head Attention (MHA):
    • Computes token interactions via Q, K, V dot-products.
    • Every token "attends" to every other token in the sequence.
  3. Stacking & Residuals:
    • Stacks many identical encoder blocks (12 for Base, 24 for Large).
    • Residual Stream: Skip-connections allow signal to flow across layers.

BERT processes text by breaking words into frequent sub-words (WordPiece).

Pre-training at Scale

BERT was the first model to leverage massive unlabeled data for general language understanding.
  • The Scale Shift: Trained on Google Books (800M words) + Wikipedia (2.5B words).
  • Masked LM (MLM): Hides 15% of tokens to predict from context.
  • Next Sentence Prediction (NSP): Learns sequence coherence.

The Result

A "foundation model" that already knows how language works and stores significant world knowledge.

Transfer Learning: Fine-tuning as "Orienting"

Transfer learning allows us to adapt a foundation model to specific IR tasks.
  • Pre-training: Learns general semantics (Expensive, done once).
  • Fine-tuning: The process of "orienting" pre-trained knowledge to a specific task (e.g., Ranking).
  • Efficiency: Requires much less data and compute than training from scratch.

Analogy

A student graduating from university (Pre-training) is "fine-tuned" when they start a specific job.

What BERT Knows: Layer by Layer

Experiments show that BERT encodes linguistic features hierarchically.
  • Lower Layers: Surface-level NLP (POS tagging, morphology, syntax).
  • Middle Layers: Complex semantics and coreference resolution.
  • Upper Layers: Abstract world knowledge and task-specific logic.

For IR Ranking

Upper layers provide the "deep" semantic matching required to bridge the lexical gap.

Cross-Encoder or MonoBERT: BERT for Re-ranking

Cross-Encoders enable deep token-level interaction between query and document.
  • Input: Feed query $q$ and document $d$ together as one sequence.
  • Interaction: Every query token attends to every document token.
  • Scoring: The final [CLS] token output serves as the relevance score.

Interpretation

Why Cross-Encoders are Accurate

Cross-Encoders enable all-to-all interaction between query and document tokens.
  • Full Interaction: Every query token attends to every document token.
  • Soft-Matching: Learned at all 12 layers of the Transformer.
  • Recall: Finds relevant docs that share zero terms with the query.

Efficiency Trade-off

Maximum accuracy comes at the cost of high latency (~500ms scale).

The Latency Bottleneck

Full token interaction makes Cross-Encoders slow for large-scale retrieval.
  • Constraint: We cannot pre-compute $P(\text{rel}|q, d)$ offline.
  • Inference: Requires running BERT for every query--doc pair at runtime.
  • Scale: Feasible only for ~100--200 candidates per query.

The Two-Stage Pipeline

Efficient neural search requires a multi-stage approach to balance cost and accuracy.
  • Recall: Sparse methods (BM25) find $\sim$1000 candidates quickly.
  • Precision: Dense methods (Cross-Encoders) re-rank the top candidates.
  • Scaling: Enables web-scale search at sub-second latencies.

The Two-Stage Pipeline (Figure)

Why Two Stages?

The pipeline separates the "wide net" retrieval from the "deep focus" ranking.
  • Sparse (Recall): Microseconds per query. Metric: Recall@1000.
  • Neural (Precision): ~5ms scale per doc. Metric: NDCG@10.
  • Combined: High precision search over an infinite corpus.

The Bounded Recall Problem

If the sparse stage misses a document, the BERT stage can never score it. The first stage determines your performance upper-bound.

The training data: MS MARCO

Neural IR models are typically fine-tuned on the massive MS MARCO benchmark.
  • Scale: 8.8M web passages and ~530k search queries.
  • Labels: Sparse; usually only one relevant passage per query.
  • Dataset: Collected from real Bing search logs and human editors.

The Neural Catalyst

This benchmark enabled the transition from lexical scoring to deep semantic ranking.

Training Pipeline: Step by Step

The training process iteratively refines the model's ability to rank relevant passages higher.
  • Triples: Train on $(q, d^+, d^-)$ tuples to learn relative relevance.
  • Loss: Optimize Contrastive or Cross-Entropy loss functions.
  • Setup: Typical training takes ~12--24 hours on one consumer GPU.

Training Pipeline: Step by Step (Figure)

Fine-Tuning BERT for Ranking

Supervised fine-tuning adapts the pre-trained model to specific ranking data.
  • Pointwise: Classify each pair (query, doc) as relevant or not.
  • Pairwise: Train so $\text{score}(q, d^+) > \text{score}(q, d^-)$.
  • Data: Usually requires large benchmarks like MS MARCO.

Common Benchmarks

  • MS MARCO: 530k queries with sparse (1 passage) labels.
  • TREC: Smaller collections with dense, multi-level labels.

MS MARCO: Sparse Labels Challenge

Since labels are sparse, models must navigate a sea of unlabeled documents.
  • False Negatives: Many relevant documents are not labeled.
  • Bias: We cannot treat all unlabeled docs as truly non-relevant.
  • Sampling: How we choose the non-relevant pairs defines model quality.

Implication

Training on random negatives is insufficient. Models need challenging, "hard" negatives.

Practice: Training BERT-base

Fine-tuning requires careful initialization and small learning rates.
  • Init: Start from pre-trained weights (e.g., bert-base-uncased).
  • Length: Must truncate query + doc to 512 sub-word tokens.
  • Schedule: Learning rate $\approx 2 \times 10^{-5}$ for ~24 hours on one GPU.

Production Tip

We are refining existing knowledge, not training from scratch!

Standard Models: monoBERT & monoT5

Two dominant architectures define the standard for neural re-ranking pipelines.
  • monoBERT: Encoder-only; uses the [CLS] token for classification.
  • monoT5: Encoder-Decoder; uses generative prompts (``Is document relevant?'').
  • Choice: monoT5 is often favored for its flexibility and accuracy.

The SOTA Baseline

Both models consistently outperform lexical methods by a large margin on TREC and MS MARCO.

Comparison: monoBERT vs. monoT5

While both are effective, they differ in architecture and training objectives.
  • monoBERT: Faster inference; trained via classification.
  • monoT5: More flexible; trained via sequence-to-sequence generation.
  • Trade-off: T5's generative approach often yields slightly higher accuracy.

Common Training Pitfalls

  • Overfitting: Neural rankers overfit MS MARCO quickly. Stop at 2--3 epochs.
  • Catastrophic Forgetting: Use small learning rates ($< 3 \times 10^{-5}$) and warmup.
  • Weak Negatives: Random samples are too easy. Use BM25 hard negatives.

The Representation Spectrum

Text representations evolved from sparse matching to deep semantic interaction.
  • Sparse: High scalability, zero semantics (Exact-match bottleneck).
  • Dense: Good semantics, high scalability via ANN (Retrieval).
  • Neural: Deepest interaction, lowest scalability (Re-ranking).
