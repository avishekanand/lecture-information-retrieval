The Remaining Problem

Cross-encoders are too slow for retrieval:
  • Re-ranking 100 docs: 0.5s ✓ Full corpus (8.8M): 12+ hours
  • BM25 misses semantic matches: ``automobile issues'' vs.\ ``car problems''
  • Bounded Recall: if retriever misses a relevant doc, the cross-encoder can never recover it

Today's Goal

The Missing Piece

Combine the semantic understanding of BERT with the retrieval speed of ANN indexes from Lecture 2.

Key Idea: The Bi-Encoder

  1. Encode all documents offline with BERT $\to$ dense vectors
  2. Store in ANN index (IVF+PQ from Lecture 2!)
  3. At query time: encode query with BERT $\to$ ANN search $\to$ top-$k$

The Trade-off

Bi-encoders are less accurate than cross-encoders (no query-document interaction) but fast enough for full-corpus retrieval.

Contrastive Learning: Core Principle

Objective

Learn embeddings such that: $$\text{score}(\vq, \vd^+) \;\gg\; \text{score}(\vq, \vd^-)$$ for relevant pair $(\vq, \vd^+)$ and irrelevant pair $(\vq, \vd^-)$.
Intuition: The embedding space is a map of meaning.
  • ``car problems'' and ``automobile issues'' $\to$ nearby points
  • ``car problems'' and ``car racing'' $\to$ far apart

InfoNCE: Noise Contrastive Estimation --- The Classification View

Key Insight

InfoNCE frames retrieval as $N$-way classification: given query $\vq$, classify which of the $N$ documents in the corpus is the relevant one, using dot-product similarity as the logit.

InfoNCE: Noise Contrastive Estimation --- The Classification View (Figure)

Theoretical InfoNCE: The Global View

The Goal: Contrastive Estimation

Given query $\vq$ and its relevant document $\vd^+$, we want to maximize the probability: $$P(\vd^+ | \vq) = \frac{\exp(\vq^\top \vd^+ / \tau)}{\sum_{\vd \in \mathcal{D}} \exp(\vq^\top \vd / \tau)}$$ where $\mathcal{D}$ is the entire corpus (e.g., 8.8M passages).

The InfoNCE Loss

The loss minimizes the negative log-probability of the positive document: $$\mathcal{L}_{\text{InfoNCE}} = -\log P(\vd^+ | \vq) = -\log \frac{\exp(\vq^\top \vd^+ / \tau)}{\sum_{\vd \in \mathcal{D}} \exp(\vq^\top \vd / \tau)}$$

InfoNCE: The Computational Bottleneck

The InfoNCE Loss

The loss minimizes the negative log-probability of the positive document: $$\mathcal{L}_{\text{InfoNCE}} = -\log P(\vd^+ | \vq) = -\log \frac{\exp(\vq^\top \vd^+ / \tau)}{\sum_{\vd \in \mathcal{D}} \exp(\vq^\top \vd / \tau)}$$

The Computational Bottleneck

The denominator requires computing the dot product for every single document in the corpus for every gradient step.
  • 8.8M documents $\times$ 128 queries per batch $\approx$ 1.1 Billion dot products.
  • Numerically and computationally impossible to do directly.

The InfoNCE Loss: Practical Approximation

In-Batch Contrastive Loss

We approximate the global sum with a sum over the current training batch of size $B$: $$\mathcal{L} = -\log \frac{\exp\!\bigl(\vq_i^\top \vp_i^+ / \tau\bigr)}{\displaystyle\sum_{j=1}^{B} \exp\!\bigl(\vq_i^\top \vp_j / \tau\bigr)}$$
Key components:
  • $B$: batch size --- provides $B-1$ free negatives per query
  • $\tau$: temperature --- controls sharpness of the softmax

InfoNCE: From $N$ Classes to $B$ Classes

Batch Approximation

Instead of classifying over the entire corpus ($N$ classes), we classify over only the batch ($B$ classes). Each batch provides $B{-}1$ in-batch negatives for free.

DPR: Architecture (Karpukhin et al., 2020)

Key Design Choices

  • Representation: [CLS] token output --- single vector per text
  • Similarity: $\text{score}(\vq, \vp) = \vq^\top \vp$ (dot product)

DPR: Architecture (Karpukhin et al., 2020) (Figure)

DPR: Training Recipe --- Loss & Negatives

Loss: InfoNCE with in-batch negatives $$\mathcal{L} = -\log \frac{e^{\vq_i^\top \vp_i^+/\tau}}{\sum_{j=1}^{B} e^{\vq_i^\top \vp_j / \tau}}$$ Negatives per query:
  1. $B-1$ in-batch negatives (free!)
  2. 1 BM25 hard negative (precomputed)

Key Insight

Mixing one BM25 hard negative with in-batch random negatives is surprisingly effective. The BM25 negative is lexically similar but not relevant --- forces the model beyond surface matching.

DPR: Why BM25 Hard Negatives?

Query: ``side effects of aspirin'' Random negative: ``The capital of France is Paris''
  • $\vq^\top \vd^- \approx 0.05$ --- trivially easy, no learning signal
BM25 hard negative: ``Aspirin was invented by Bayer in 1899''
  • Contains ``aspirin'' $\to$ BM25 ranks it high
  • But not about side effects $\to$ not relevant
  • $\vq^\top \vd^- \approx 0.6$ --- confusing, strong learning signal!

The Principle

The model must learn: ``just because a passage mentions aspirin doesn't mean it's about side effects.'' This is exactly what BM25 cannot do.

A single 768-dim vector for the entire document? That's a severe information bottleneck. Can we do better?

The Single-Vector Bottleneck

Problem with DPR

The entire document is compressed into a single 768-dim vector.
Consequences:
  • Cannot capture fine-grained token-level alignments
  • ``Who founded Wikipedia?'' vs.\ ``When was Wikipedia founded?'' $\to$ similar scores!
  • Long documents lose information

Solution

Keep all token embeddings, not just [CLS]. Compare at the token level.

ColBERT: Architecture (Khattab & Zaharia, 2020)

Key Idea

Encode query and document separately (like bi-encoder), but compare at the token level using MaxSim.

Late Interaction

Documents are encoded offline (token-level embeddings stored). The ``interaction'' happens late --- only at query time via MaxSim.

ColBERT: Architecture (Khattab & Zaharia, 2020) (Figure)

ColBERT: MaxSim Scoring

MaxSim Definition

$$\text{MaxSim}(\vq, \vd) = \sum_{i=1}^{|\vq|} \max_{j=1}^{|\vd|}\; \cos(\vq_i, \vd_j)$$
Step by step:
  1. For each query token $\vq_i$, find the document token $\vd_j$ with highest cosine similarity
  2. Take that maximum similarity
  3. Sum across all query tokens
Properties:
  • Asymmetric: Every query token must find a match; document tokens can be ignored
  • Decomposable: Document embeddings precomputed and stored offline
  • Token embeddings: $\vq_i, \vd_j \in \mathbb{R}^{128}$ (projected down from 768)

Worked Example: MaxSim Scoring

Query: ``who founded wikipedia'' (3 tokens) Document: ``wikipedia was founded by jimmy wales'' (6 tokens)

Hard vs.\ Easy Negatives: Why It Matters

The Evolving Encoder and Negative Quality

Key Insight: What Counts as ``Hard'' Changes Over Training

The encoder's understanding of relevance evolves across training epochs. This changes which negatives are informative:
Early training: The encoder has no notion of relevance.
  • BM25 negatives (lexically similar but irrelevant) are not yet hard --- the model cannot even distinguish basic relevance.
  • Random negatives provide sufficient learning signal.
Later training: The encoder understands basic relevance.
  • Random negatives become trivially easy --- zero gradient.
  • BM25 negatives now become genuinely hard --- the model must learn to distinguish lexical overlap from true relevance.

The Evolving Encoder: Representations Shift Over Training

Implication

The optimal negative sampling strategy is not fixed --- it should adapt to the encoder's current capability.

The Evolving Encoder: Representations Shift Over Training (Figure)

From Static to Dynamic Negatives

Static Negatives (e.g., BM25)

Negatives are chosen once, before training and remain fixed:
  • Cheap to compute (run BM25 once).
  • Eventually become too easy as the encoder improves.

Dynamic Negatives (e.g., ANCE)

Negatives are defined by the current encoder state:
  • A hard negative = a document the model currently scores high, but is not the ground-truth relevant document.
  • These are the most confusing examples {right now} --- maximally informative gradients.
  • Requires periodically re-encoding the corpus and refreshing the index.
Static: fixed before training $\longrightarrow$ Dynamic: adapts with the model

ANCE: Asynchronous Index Refresh

Core Innovation

Maintain an ANN index during training. Use the model's own top-$k$ nearest passages as hard negatives.

Key Finding

Combining BM25 and dense retrieval outperforms either alone.
$$\text{score}_{\text{hybrid}}(q, d) = \alpha \cdot \text{score}_{\text{BM25}}(q,d) + (1-\alpha) \cdot \text{score}_{\text{dense}}(q,d)$$

Why Hybrid Works

Sparse captures exact matches (named entities, rare terms). Dense captures semantic similarity (paraphrases). They are complementary, not competing.

Score-Agnostic Fusion: Reciprocal Rank Fusion

The Challenge

Dense scores (dot products) and Sparse scores (BM25) are on different scales. How to combine them without fragile hyperparameter tuning?

Reciprocal Rank Fusion (RRF)

RRF relies only on rankings. It favors documents that appear high in any characteristic (sparse or dense).

Beyond Ranking: Retrieval-Augmented Generation (RAG)

The Paradigm Shift

In modern AI applications, retrieval is not the final step. It is the foundation for Large Language Models (LLMs).
  • Problem: LLMs hallucinate and have a fixed knowledge cutoff.
  • Solution: Retrieve relevant, up-to-date passages and provide them as grounding context in the model's prompt.
  • Impact: Retrieval expands the model's memory and enables it to cite its sources.

The Modern IR Pipeline with RAG

Retrieval as the "Eyes" of the LLM

If the retriever (bi-encoder) or re-ranker (cross-encoder) fails, the generator cannot produce a factual answer. High-quality IR is the bottleneck for RAG accuracy.

Key Takeaways

1. Bi-Encoders Enable Semantic Retrieval at Scale

Encode documents offline, search with ANN indexes from Lecture 2. Solves the vocabulary mismatch problem that BM25 cannot.

2. Contrastive Learning Is the Training Paradigm

InfoNCE loss with in-batch negatives. Larger batches = more negatives = better training. Temperature $\tau$ is a critical hyperparameter.

3. DPR $\to$ ColBERT $\to$ ANCE: Improving Dense Retrieval

DPR: single-vector bi-encoder. ColBERT: token-level late interaction. ANCE: dynamic hard negatives via asynchronous index refresh.

4. Hybrid Retrieval Is the Practical Standard

Dense + sparse complementary. BM25 for exact match, dense for semantics. Cross-encoder on top for final precision.
