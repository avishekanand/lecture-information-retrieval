The Inverted Index: Still Relevant in 2024

What Powers These Systems?

  • Google Search
  • Elasticsearch / OpenSearch
  • Database full-text search (PostgreSQL, MySQL)
  • RAG sparse retrieval component
Answer: Inverted indexes
Why still relevant in the age of neural networks?
  1. Exact matching: Find "iPhone 15 Pro Max" precisely
  2. Speed: 5-10ms queries on millions of documents
  3. Explainability: Can see which terms matched
  4. Hybrid search: Complements dense retrieval

Inverted Index: Core Structure

Documents:
  • Doc1: ``neural networks for deep learning''
  • Doc2: ``deep learning applications''
  • Doc3: ``neural machine translation''
Query: ``neural learning'' Process: Lookup ``neural'' $\rightarrow$ [1, 3], lookup ``learning'' $\rightarrow$ [1, 2] Intersection: [1] $\leftarrow$ Doc 1 contains both terms

Inverted Index as a Sparse Matrix

Vector Space Interpretation

  • Documents = Rows ($D$)
  • Terms = Columns ($V$)
  • Values = Weights (TF-IDF, BM25)

Key Insight

The Inverted Index is a Compressed Sparse Column (CSC) storage of the Document-Term matrix. We avoid storing zeros!

Why Inverted Index is Fast

Naive Approach:
  1. Scan all documents
  2. Check if query terms present
  3. Rank matches
Complexity: $O(N \cdot M)$
  • $N$ = number of documents
  • $M$ = avg document length
For 10M docs, 500 words each: 5 billion comparisons!

Inverted Index:
  1. Lookup term in dictionary
  2. Fetch posting list
  3. Merge posting lists
Complexity: $O(k \cdot p)$
  • $k$ = query terms (typically 2-5)
  • $p$ = avg posting list size
For query ``neural learning'': Lookup 2 terms, merge ~1000 docs $\rightarrow$ 1000$\times$ faster!

Posting List Formats: Evolution

Level 1: DocID Only (Boolean Retrieval)

neural: [1, 3, 5, 12, 27, 89]
  • Supports: AND, OR, NOT queries
  • Cannot: Rank by relevance

Level 2: With Frequencies (Ranked Retrieval)

neural: [(1, tf=2), (3, tf=1), (5, tf=3), (12, tf=1), ...]
  • Supports: TF-IDF, BM25 scoring
  • This is what we use in practice

Level 3: With Positions (Phrase Queries)

neural: [(1, tf=2, pos=[5,42]), (3, tf=1, pos=[8]), ...]
  • Supports: ``neural networks'' (exact phrase)
  • Enables: Proximity scoring
  • Cost: 2-3$\times$ larger index

Text Preprocessing: Key Decisions

Stemming (Porter):
  • running $\rightarrow$ run
  • university $\rightarrow$ univers
  • Fast, crude

Lemmatization:
  • running $\rightarrow$ run
  • better $\rightarrow$ good
  • Accurate, slower

Stop Words: Modern Perspective

Traditional View (pre-2010):
  • Remove common words: the, a, an, is, are, of, in, ...
  • Smaller index
  • Less noise
Problem:
  • ``to be or not to be'' $\rightarrow$ ``''
  • ``The Who'' (band) $\rightarrow$ ``Who''
  • Loses phrase meaning

Modern View (2010+):
  • Keep stop words
  • Storage is cheap
  • Important for phrases
  • Neural models need them
When to remove:
  • Very large scale (web search)
  • Memory-constrained devices
  • Proven unnecessary for domain

Index Compression: Why It Matters

The Bottleneck

Modern CPUs: 3+ GHz, billions of operations/second Modern SSDs: 500 MB/s read speed Decompression: 2-5 GB/s Reading uncompressed: 0.5 GB/s $\rightarrow$ Reading compressed data is faster!

Real Impact

Collection: 10M documents, 100 words each Uncompressed index: 10 GB Compressed (5$\times$): 2 GB Benefits:
  • More index fits in RAM (faster queries)
  • Lower storage costs
  • Faster I/O (disk $\rightarrow$ memory)

Gap Encoding: The Key Insight

Original posting list (docIDs): [5, 127, 214, 215, 512, 1024, 2048, 10239] Observation: Most gaps between consecutive docIDs are small Store gaps instead: [5, 122, 87, 1, 297, 512, 1024, 8191]

Why This Helps

Original: Numbers range 5-10,239 $\rightarrow$ need 14 bits each Gaps: Most numbers small ($<$ 256) $\rightarrow$ need fewer bits Small numbers compress much better!

Decoding

Sequential only: $\text{docID}_i = \text{docID}_{i-1} + \text{gap}_i$

Variable Byte (VB) Encoding

Core Idea

Use 1 byte for small numbers, multiple bytes for large numbers
  • 7 bits for data
  • 1 bit (MSB) for continuation: 1 = last byte, 0 = more bytes follow
NumberBinaryVB Encoding
51011_0000101 (1 byte)
12211110101_1111010 (1 byte)
82411001110000_0000110 1_0111000 (2 bytes)
Example compression:
  • Gaps: [5, 122, 87, 1] $\rightarrow$ 4 bytes (VB encoded)
  • Uncompressed: 16 bytes (4 $\times$ 32-bit integers)
  • Compression ratio: 4$\times$

VB Encoding: Step-by-Step Example

Encode 824:
  1. Binary representation: 1100111000 (10 bits needed)
  2. Needs 2 bytes (10 $>$ 7)
  3. Split into 7-bit chunks from right:
    • Lower 7 bits: 0111000
    • Upper bits: 0000110
  4. First byte (higher bits): 0_0000110 (continuation = 0)
  5. Second byte (lower bits): 1_0111000 (continuation = 1, last byte)
  6. Result: 00000110 10111000
Decode:
  1. Read bytes until continuation bit = 1
  2. Strip continuation bits: 0000110 0111000
  3. Concatenate: 0000110_0111000 = 1100111000 = 824

Compression in Practice

Typical Results

  • VB encoding: 3-4$\times$ compression
  • More sophisticated methods (PForDelta): 6-10$\times$ compression
  • Dictionary compression (front coding): 30-50

Production Systems

  • Lucene/Elasticsearch: Uses FOR (Frame of Reference) with SIMD
  • Google: Custom compression optimized for their hardware
  • Your projects: VB encoding is good enough (simple, effective)

Key Takeaway

Compression is not optional --- it makes systems faster and cheaper

The ANN Search Problem

Task

Given:
  • Collection of $N$ vectors: $\{\vec{d}_1, \vec{d}_2, \ldots, \vec{d}_N\}$
  • Query vector: $\vec{q}$
Find: $k$ vectors most similar to $\vec{q}$ (typically cosine similarity)

Naive Solution: Exact Search

Compare query to all $N$ vectors: $O(N \cdot d)$ For $N = 10^9$ docs, $d = 768$ dimensions: 768 billion operations $\rightarrow$ too slow! ($>$1 second even on GPU)

Solution: Approximate Nearest Neighbor (ANN)

Trade exactness for speed: find approximately top-$k$ Typical accuracy: 90-99% recall (good enough for IR!)

ANN: Accuracy-Speed Trade-off

Typical operating points:
  • High quality: 95-99% recall @ 100-500 QPS (production RAG)
  • Balanced: 90-95% recall @ 500-2000 QPS (most use cases)
  • Fast: 80-90% recall @ 2000-10,000 QPS (first-stage retrieval)

The Standard ANN Approach: IVF + PQ

  1. Clustering-Based (IVF)
    • Cluster vectors with k-means
    • Search only in nearest clusters
    • Fast, scalable
    • Lower accuracy without tuning
  2. Product Quantization (PQ)
    • Compress vectors aggressively (32$\times$)
    • Approximate distances in compressed space
    • Very memory-efficient
    • Moderate accuracy loss
  3. Combined (IVF + PQ)
    • Use clustering to narrow search area
    • Use compression to fit index in RAM and speed up distance calculation
    • The industry standard for large-scale RAG

IVF: Inverted File Index (for Vectors!)

Core Idea

Cluster vectors, then search only in nearest clusters

IVF: Inverted File Index (for Vectors!) (Figure)

IVF: Construction and Search

Construction

[1] Run k-means on all vectors $\rightarrow$ get $k$ cluster centers (e.g., $k=1000$) Assign each vector to its nearest cluster center Build inverted lists: cluster $i$ $\rightarrow$ [vector IDs in cluster $i$]

Search

[1] Input: Query $\vec{q}$, $n_{\text{probe}}$ (clusters to search), $k$ Compute distance from $\vec{q}$ to all cluster centers Select $n_{\text{probe}}$ nearest cluster centers $\text{candidates} \gets []$ {each selected cluster} Fetch all vectors in cluster Compute exact distances to $\vec{q}$ Add to candidates Sort candidates by distance, return top-$k$
Key parameter: $n_{\text{probe}}$ (1 = fast/low recall, 100 = slow/high recall)

Product Quantization: Compression

Goal

Compress 768-dim vector (3072 bytes) to 96 bytes (32$\times$ compression!)
Method:
  1. Split each vector into $m$ subvectors (e.g., $m=96$ subvectors of 8 dims)
  2. Quantize each subvector: assign to nearest centroid (256 centroids)
  3. Store centroid ID (1 byte) instead of full subvector

Example

Original: [0.23, -0.41, 0.87, 0.12, -0.34, 0.56, 0.78, -0.23, ...] (768 floats) Split into 96 subvectors:
  • Subvec 1: [0.23, -0.41, 0.87, 0.12, -0.34, 0.56, 0.78, -0.23] $\rightarrow$ Centroid 42
  • Subvec 2: [0.45, 0.67, -0.12, 0.89, 0.34, -0.56, 0.23, 0.11] $\rightarrow$ Centroid 123
  • ... (96 total)
Compressed: [42, 123, 7, 89, ...] (96 bytes)

Product Quantization: Distance Approximation

Key Insight

Can approximate distances using only centroid IDs (no need to decompress!)
Algorithm:
  1. Precompute distance table: query subvector to all centroids
    • For each of 96 subspaces
    • For each of 256 centroids
    • Total: 96 $\times$ 256 = 24,576 distances (computed once per query)
  2. For each document:
    • Approximate distance = $\sum_{i=1}^{96} \text{distTable}_i[\text{code}_i]$
    • Just 96 lookups + additions (very fast!)

Performance

  • Memory: 32$\times$ compression (3GB $\rightarrow$ 96MB for 1M vectors)
  • Speed: 10-100$\times$ faster than exact distance
  • Accuracy: 80-95% recall typical

IVF + PQ: The Production Standard

Best of Both Worlds

Combine IVF (clustering) with PQ (compression)
Architecture:
  1. Offline: Cluster vectors into $k$ clusters (IVF)
  2. Offline: Compress vectors with PQ
  3. Online: Find $n_{\text{probe}}$ nearest clusters
  4. Online: Search within those clusters using PQ distances

Real-World Impact

Facebook/Meta AI Similarity Search (FAISS):
  • 1 billion vectors $\times$ 768 dims $\times$ 4 bytes = 3 TB
  • With IVFPQ: 1 billion vectors $\times$ 64 bytes = 64 GB
  • 50$\times$ memory reduction!
  • Still achieves 90-95% recall with proper tuning
Used by: Meta, OpenAI, many large-scale systems

  • Use Sparse when you need to find exact names, model numbers, or unique terms.
  • Use Dense when users ask questions in natural language or use synonyms.

Case Study: Academic Paper Search (100M+)

Scenario

Research search engine with 100M papers (arXiv, PubMed, etc.)
The Challenge:
  • Cannot hold 100M raw vectors in RAM (300GB+).
  • Users need both exact titles (Sparse) and broad conceptual search (Dense).

The Strategy: IVF+PQ

  • Sparse (BM25): Index title, authors, abstracts for keyword matching.
  • Dense (IVF+PQ):
  • Cluster into 10k centroids (high speed).
  • Compress vectors 32$\times$ (fits on a single server!).
Result: Billion-scale search with low latency and low cost.

Common Pitfalls and Solutions

Pitfall 1: Using Only Dense for Entity-Heavy Domains

Problem: Query ``Python 3.11 release date'' $\rightarrow$ matches ``snake release from zoo'' Solution: Use hybrid! BM25 catches ``Python 3.11'' exactly

Pitfall 2: Not Tuning ANN Parameters

Problem: Using default ef_search=16 $\rightarrow$ only 80% recall Solution: Tune on dev set. Start ef_search=64, increase until recall plateaus

Pitfall 3: Inconsistent Preprocessing

Problem: Index with stemming, query without $\rightarrow$ no matches! Solution: Same preprocessing for both. For embeddings: no stemming!

Pitfall 4: Forgetting to Normalize Vectors

Problem: Using unnormalized embeddings $\rightarrow$ Euclidean distance $\neq$ cosine similarity Solution: Always normalize: faiss.normalize_L2(embeddings)

Key Takeaways

1. Inverted Indexes are the Backbone

Keyword matching is essential for exact queries and entities. Don't throw away BM25!

2. Dense Retrieval Enables Semantic Search

Embeddings capture meaning, but require ANN techniques like IVF+PQ to work at scale.

3. Candidate Generation is the Bottleneck

Fast indexing determines how many documents your re-rankers or LLMs can "see".

4. Scalability requires Compression

Techniques like Product Quantization (PQ) are vital for fitting large indexes in RAM.
